MySQL基础

一、数据库

1.1 数据库概念

数据库（DataBase）是长期存储在计算机内部有结构的，大量的，共享的数据集合

  • 长期存储：持久存储
  • 有结构：
    • 类型：存储的数据是有类型的
    • 关系：存储数据于数据之间的关系
  • 大量：大多数数据库是文件系统的，也就是说存储在数据库中的数据实际上存储在磁盘的文件中
  • 共享：多个应用程序通过数据库实现数据共享

1.2 关系型数据库与非关系型数据库

  • 关系型数据库

采用了关系模型来组织数据的存储，以行和列的的形式存储数据并记录数据与数据之间的关系
将数据存储在表格中，可以通过建立表格与表格之间的关系来维护数据与数据之间的关系

  • 非关系型数据库

采用键值对的模型来存储数据，只完成数据的记录，不会记录数据与数据之间的关系
基于其特定的的存储结构来解决一些大数据应用难题

二、SQL

2.1 SQL概述

SQL (Structured Query Language) 结构化查询语言，用于存储、查询、更新数据及管理关系型数据库系统

2.1.1 SQL分类

根据SQL指令完成的数据库操作不同，可以将SQL指令分为四类

  • DDL (Data Define Language) 数据库定义语言
    • 用于完成对数据库对象（数据库，数据表，视图，索引等）的管理
  • DML (Data Manipulation Language) 数据库操作语言
    • 用于完成对数据表中的数据进行添加，删除，修改等操作
  • DQL (Data Query Language) 数据库查询语言
    • 用于将数据表中的数据查询出来
  • DCL (Data Control Language) 数据库控制语言
    • 用于完成事务管理等操作

2.2 SQL基本语法

  • sql指令不区分大小写
  • 每条sql指令结束后都以；结尾
  • sql关键词之间以空格进行分割
  • sql之间可以不限制换行

五、存储过程

5.1 存储过程介绍

5.1.1 SQL指令执行过程

5.1.2 存储过程介绍

将能够完成特定功能的sql指令进行封装（sql指令集），编译后存储在数据库服务器上，并且为之取一个名字，
客户端可以通过这个名字直接调用这个sql指令集，获得执行结果

优点

  • sql指令无需客户端编写，通过网络传送，既可以节省网络开销，同时避免sql指令在网络传输过程中被恶意篡改
    保证安全性
  • 存储过程经过编译创建并保存在数据库中的，执行过程无需重复的进行编译操作，对sql指令的性能进行提升
  • 存储过程中多个sql指令之间存在逻辑关系，支持流程控制语句，可以实现更为复杂的业务

缺点

  • 存储过程时针对不同数据库编写的，当我们切换到其他数据库就要重新进行编写
  • 存储过程受限于数据库产品
  • 如果是高并发业务会增加执行时间

六、触发器

6.1 触发器概念

触发器是一种特殊的存储过程。触发器和存储过程一样是一个能够完成特定功能，存储在数据库服务器上的SQL
片段，但触发器无需调用，当对数据表中的数据执行DML操作时会自动触发SQL片段的执行，而无需手动调用

在mysql中，只有执行insert/delete/update操作才能触发触发器的执行

6.4 触发器使用总结

6.4.1 优点

  • 触发器是自动执行的,当对触发器相关表执行相应DML操作会立即触发触发器
  • 触发器可以实现表中的级联操作(关联操作),有利于保证数据的完整性
  • 触发器可以对DML操作的数据更为复杂的合法性进行校验

6.4.2 缺点

  • 使用触发器实现的业务逻辑如果出现问题将难于定位,后期维护困难
  • 如果大量使用触发器,导致代码结构杂乱,增加程序复杂性
  • 当触发器处操作的数据量比较大时,执行效率会降低

6.4.3 使用建议

  • 在互联网项目中,应避免使用触发器
  • 对于并发量不大的项目可以选择使用存储过程,但在互联网使用存储过程
    • 原因是存储过程将实现业务逻辑交给数据库处理,增加了数据库负载,不利于数据库迁移

7.4 视图数据的特性

视图是虚拟表,查询视图的数据是来源于数据表的.当对视图数据进行操作时,对原数据表的数据是否有影响呢?

查询操作:如果在数据表中添加了新的数据,且这个数据满足创建视图时查询语句的条件,
通过查询视图也可以查询出新增的数据,当删除原表中满足查询条件的数据时,
也会从视图中删除.

新增数据: 如果在视图中添加数据,那么数据会被添加到原表

删除数据: 如果从视图删除数据也将从原表中删除

修改数据: 如果在视图中修改数据原表也会修改

==视图的使用建议==: 对复杂查询简化操作,并且不会对数据进行修改的情况下可以使用视图,如果是多表联合查询,则不会成功

八、索引

数据库是用来存储数据的,在互联应用中数据库中存储的数据会很多(大数据),数据表中数据的查询速度会随着
数据量的增长逐渐变慢,从而导致相应用户请求的速度变慢—-用户体验很差,如何提高数据的查询效率呢?

8.1索引介绍

索引,就是将数据表中的某一列或某几列的值取出来构造成B+树结构进行存储,生成数据表的==目录==

当我们进行数据查找时现在目录中进行查找,得到对应的数据地址,然后再到数据表中根据地址快
速获取数据记录避免全表扫描

8.2 索引的分类

mysql中的索引,根据创建索引的列的不同分为:

  • 主键索引 : 在数据表的主键字段创建的索引,这个字段必须被primary key修饰,每张表只能有一个主键
  • 唯一索引 : 在数据表中的唯一列创建的索引,此列的值只能出现一次,可以为null
  • 普通索引 : 在普通字段上添加的索引,没有唯一限制
  • 组合索引 : 两个及以上字段联合起来创建的索引

==说明== :
1.在创建表时,将字段声明为主键,会自动在主键字段创建主键索引
2.在创建表时,将字段声明为唯一键(unique),会自动在唯一字段创建唯一索引

8.7 索引的使用总结

8.7.1 优点

  • 索引大大降低了数据库服务器在执行查询操作时扫描的数据量,提高查询效率
  • 索引可以避免服务器进行排序、将随机的IO变成顺序IO
  • 加锁

8.7.2 缺点

  • 索引是根据数据表列的值创建的,当数据表中数据发生DML操作时,索引页会更新
  • 索引文件也会占用磁盘空间

8.7.3 使用注意事项

  • 数据表中数据不多时,全表扫描可能更快,不建议使用索引
  • 数据量很大,但DML操作频繁,不建议使用索引
  • 不要在数据重复度高的列上创建索引(性别)
  • 创建索引后要注意sql语句的编写,避免索引失效

九、数据库事务

9.1 数据库事务介绍

  • 把完成特定业务的多个数据库DML操作步骤称之为一个事务
  • 事务,就是完成同一个业务的多个DML操作
  • 比如,借书和修改库存

9.2 数据库事务特性(ACID)

  • 原子性(Atomicity) : 一个事务中的多个DML操作,要么同时执行成功,要么执行失败
  • 一致性(Consistency) : 事务执行前后的数据库中的数据是一致的,完整性和一致性不被破坏
  • 隔离型(Isolation) : 数据库允许多个事务同时执行,多个并行的事务不能相互影响.
  • 持久性(Durability) : 事务完成后会数据的操作是永久的

9.3 MySQL事务管理

9.3.1 自动提交与手动提交

  • 在mysql中,默认DML指令的执行是自动提交的,当我们执行一个DML指令后,自动同步到数据库中

9.3.2 事务管理

开启事务,就是关闭自动提交

  • 在开始事务的第一个操作之前,执行 start transaction 开启事务
  • 依次执行事务中每个DML操作
  • 如果在执行过程中任何步骤出现异常,则执行 rollback 回滚事务
  • 如果事务所有DML操作都成功,执行 commit 提交事务

9.4 事务隔离级别

数据库允许多个事务并行,多个事务之间是隔离的、相互独立的;如果多个事务之间不隔离,操作同一数据时
可能会导致数据的一致性被破坏

mysql数据库事务隔离级别

  • 读未提交(read uncommitted) : t2可以读取t1执行但未提交的数据,可能会导致脏读

脏读: 一个事务读取到了另一个事务中未提交的数据,因为未提交的数据可能会回滚

  • 读已提交(red committed) : t2只能读区t1已经提交的数据,避免了脏读,但可能会导致虚读

不可重复读(虚读) : 同一个事务中两次查询到的价格不一致

例如: t2查询数据后, t1 修改了数据,t2第二次查询的数据与第一次查询的数据不一致

  • 可重复读(repeatable read) : t2执行第一次查询后,在事务结束之前其他事务不能修改对应数据,
    避免了虚读,但可能会导致幻读

幻读: t2把所有商品价格修改为1元,然后t1向表中新增一个3元的商品,t2再查询时会以为查询出
了与修改不一致的商品

  • 串行化(serializable) : 同时只允许一个事务对数据表进行操作;避免了脏读,虚读,幻读问题
隔离级别 脏读 不可重复读(虚读) 幻读
Read uncommitted yes yes yes
Read committed no yes yes
Repeatable read no no yes
Serializable no no no

五、SQL注入问题

5.1 什么是SQL注入

在JDBC操作sql指令编写过程中,如果sql指令需要数据,可以通过字符串拼接形式拼接到指令中
如 : String sql = “delete from books where book_id =”+s;
这种形式可能导致==因变量值的改变导致SQL指令原意发生改变==,被称为sql注入

例如:

  1. 如果s为1 ; sql为: delete from books where book_id where book_id =1;
  2. 如果s为 1 or 1=1 ; sql 为: delete from books where book_id where book_id =1 or 1=1;

5.2 解决sql注入问题

使用PreparedStatement进行SQL预编译解决sql注入问题

  1. 如果sql指令需要参数,一律使用==?==参数占位符

  2. 如果sql指令有==?==,在jdbc操作步骤中,从connection对象获取preparedStatement对sql进行预编译

    String sql = “delete from books where book_id=?”;
    PreparedStatement statement = connection.prepareStatement(sql);

  3. 预编译完成后,通过PreparedStatement对象给预编译后sql指令的==?==赋值

  4. sql指令中的==?==完成赋值之后,通过preparedStatement对象执行sql,但不再加载,因为加载过了
    int i = statement.executeUpdate();

十、数据库连接池

10.1 什么是数据库连接池

如果每个JDBC操作都需要数据库连接且重新创建,用完后都销毁,JVM会因频繁的创建、销毁连接而占
用额外的系统资源.

数据库连接本质上是可被重用的资源(当一个JDBC操作完成后,其创建的连接可被其他JDBC操作使用)

  • 基于这个特性,可以创建一个存放数据库连接的容器(连接池),数据库连接池是有最大容量的,当进行
    JDBC操作时,不用创建连接,直接从这个容器中获取连接

    • 如果容器中没有数据库连接,或没有空闲的数据库连接,则创建新的数据库连接存入连接池并给
      这个操作使用,使用完成后无需关闭连接,直接归还到这个容器中即可

    • 如果容器中有空闲连接,则无需创建新连接,直接从容器中获取连接并使用

    • 如果容器中没有空闲的连接,且连接池数量已经达最大值,当前操作就会等待.等待连接池某个连
      接被归还,归还后再使用

连接池:存放数据库连接对象的容器

连接池作用: 对数据库连接进行管理,减少因重复创建、销毁数据库连接导致的系统开销

十二、Apache DBUtils

12.1 DBUtils介绍

Commons Utils 是Apache 组织提供的一个针对JDBC进行简单封装的开源工具类库,使用DBUtils可以极大简化
JDBC应用程序开发,同时不会影响数据库的性能.

DBUtils是java编程中数据库访问的轻巧,实用的工具类库

  • 提供对数据表通用的DML操作
  • 提供对数据表通用的DQL操作(可以把结果封装成对象,集合等类型)

DBUtils中的核心类:

  • QueryRunner, 用于执行SQL指令(相当于我们自己封装的CommonDAO类)
    • update( )用于执行,DML指令
    • query( ) 用于执行DQL指令
  • ResultSetHandler接口,结果集处理,相当于我们自己封装的RowMapper接口
    • BeanHandler实现类,当查询语句返回一条结果时,它将结果转换成一个对象
    • BeanListHandler实现类,当查询语句返回多条结果时,它将结果转换成一个集合
    • ScalarHandler实现类,当查询语句返回一个值时,它将结果转换为java变量
